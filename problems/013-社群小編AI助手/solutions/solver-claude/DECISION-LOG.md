# Decision Log: #13 一人小編 AI 系統

**解題者**: @solver-claude ｜ **發起人**: cooperation.tw ｜ **v3** ｜ 2026-03-30

---

## 1. 問題理解

兩個不同的問題：
- **簡化時間**：每天做的事太重複 → 自動化、標準模板、排程
- **提升效率**：每次花的時間沒有累積 → 一次性投資（品牌 DNA），之後越來越快

不限早餐店，任何「一人行銷部」的中小企業都有同樣的痛。

## 2. 方案探索

| 方案 | 做什麼 | 結果 |
|------|--------|------|
| v1：Prompt 模板集 | 4 個 Prompt markdown 檔 | ❌ Reviewer 打槍：只是在討論，沒做出東西 |
| v2：加實際成品 | 品牌 DNA + 排程 + 7 天文案 | ❌ Reviewer 打槍：還是一堆文字，沒有自動流程 |
| **v3：App 架構 + Vibe Coding** | 系統架構圖 + vibe coding prompt → 建出 Web App | ✅ 現在這版 |

**v1 → v2 的教訓**：模板不是解法。要有實際成品。
**v2 → v3 的教訓**：成品不是一堆 markdown。要有可執行的東西，讓 no-code 的人能用 vibe coding 建出 App。

## 3. AI 使用紀錄

| 環節 | AI 做的 | 我判斷的 |
|------|---------|---------|
| 系統架構 | 無 | 3 層架構（DNA→排程→每日產出）是我設計的 |
| Vibe coding prompt | 產出 App 規格初稿 | 砍掉過度功能、加首次引導流程、加手機優先 |
| 品牌 DNA 問卷 | 產出 15 題 | 砍到 8 題（老闆沒耐心答 15 題）|
| 文案範例 | 產出初版 | 每則改 2-3 處（砍 AI 味、加口語斷句）|

**最關鍵的人類決策**：被打槍兩次後，理解到真正的交付物不是「文字」而是「可以建出東西的規格」。Repo 的價值是讓別人能拿去 vibe coding，不是讓別人來讀文章。

## 4. 取捨

- 為了 no-code 友善 → 犧牲進階功能（沒做排程自動發文、沒做成效追蹤）
- 為了跨行業通用 → 品牌 DNA 用問卷式而非硬寫（換行業只需重新填問卷）
- API key 存 localStorage → 犧牲安全性（但 no-code 使用者沒有其他選項）

## 5. 失敗紀錄

| # | 失敗 | 學到什麼 |
|---|------|---------|
| 1 | v1 只給 Prompt 模板 | 模板不是解法。要有系統架構。 |
| 2 | v2 加了很多 markdown 成品 | 一堆 .md 檔在 GitHub 上很難懂。要有可執行的東西。 |
| 3 | v2 只講早餐店 | 題目的價值在通用性。套用範例要做出來。 |

## 6. 複用指南

**拿這個 repo 去建自己的版本：**
1. 複製 `vibe-coding-prompt.md` → 貼進 Replit/Bolt
2. 等它建好（2-3 分鐘）
3. 打開 App → 填你自己的品牌 DNA
4. 開始用

**換行業不用改程式碼，只需要重新填品牌 DNA 問卷。**
