# 評審指南 (Reviewer Guide)

## 你的影響力

你的 30 分鐘 Review，對學生的價值可能超過一整學期的課。

在 SolveTW，你的 Review：
- 直接影響解題者的聲望積分（認證業界主管的權重是一般用戶的 **10 倍**）
- 可能是學生第一次被業界專業人士認真看過作品
- 你的評語會成為學生改進的指南

---

## Review 流程

1. 收到 PR 通知（GitHub Email 或平台通知）
2. 閱讀解法的 **Decision Log**（最重要）
3. 看程式碼/Demo（評估實作品質）
4. 在 PR 上留下評分 + 評語
5. 完成（通常 30-60 分鐘）

---

## Review Rubric

### 評分維度（每項 1-5 分）

| 維度 | 5 分 (Exemplary) | 3 分 (Solid) | 1 分 (Weak) |
|------|-----------------|-------------|-------------|
| **問題理解** | 精準抓到核心挑戰，有獨到洞察 | 正確理解題意 | 只是複述題目 |
| **方案選擇** | 3+ 方案深入比較，選擇理由充分 | 2 方案基本比較 | 只寫選的方案 |
| **AI 透明度** | 清楚區分 AI/人類貢獻，有關鍵決策點 | 列出工具，基本區分 | 只說「用了 AI」 |
| **技術實作** | 可運行、架構清晰、有錯誤處理 | 可運行，基本功能完整 | 有明顯 bug 或無法運行 |
| **Decision Log 品質** | 六欄位完整、內容深入、失敗紀錄真實 | 欄位完整、內容充實 | 欄位缺漏或內容空洞 |
| **實用性** | 可直接用或稍改即可用 | 概念可行 | 離實用差距很大 |

### 總分判定

| 平均分 | 判定 | 動作 |
|--------|------|------|
| 4.0-5.0 | **Approve + Featured 推薦** | 進入 Archive，考慮推薦 |
| 3.0-3.9 | **Approve** | 進入 Archive |
| 2.0-2.9 | **Request Changes** | 給改進建議，學生修改後重新 Review |
| 1.0-1.9 | **Reject** | 給具體原因和方向 |

## Review 心法

### 重點看 Decision Log，不只看 Code

程式碼 AI 都能寫。你要看的是：
- **他有沒有自己的思考？** — 還是只是按 AI 說的做
- **他知不知道自己方案的弱點？** — 取捨紀錄是否誠實
- **他會不會從失敗中學習？** — 失敗紀錄的品質
- **他能不能清楚溝通？** — Decision Log 的表達能力

### 建設性 > 批判性

這些學生可能是第一次被業界看作品。你的回饋方式很重要：

**好的回饋**：
> 「你的方案選擇理由很清楚。建議在錯誤處理部分加上 retry 邏輯，
> 因為真實環境中 API 有時會 timeout。這在我們公司是很常見的問題。」

**不好的回饋**：
> 「錯誤處理太差。」

### 帶入業界觀點

你最獨特的價值是業界經驗。回饋時加入：
- 「在我們公司，這個場景會...」
- 「業界通常的做法是...」
- 「這個方案在實際部署時要注意...」

## 常見問題

### 「我很忙，Review 要花多少時間？」

一份解法通常 30-60 分鐘。如果你只有 30 分鐘：
1. 先看 Decision Log（15 分鐘）
2. 快速看 Demo/程式碼（10 分鐘）
3. 填評分表 + 簡短評語（5 分鐘）

### 「學生的解法品質很差怎麼辦？」

給建設性的回饋，指出具體的改進方向。
標記為 Request Changes，讓學生有機會改進。

### 「我想招人，可以直接聯繫解題者嗎？」

在 Review 模板最後勾選「想聯繫」，平台會在解題者同意後媒合。
不要在 PR 公開留言中要求個人資訊。

### 「我也想出題」

太好了！你可以同時是 Reviewer 和 Problem Owner。
用 [出題表單](../../issues/new?template=new-problem.yml) 提交題目。
