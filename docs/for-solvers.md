# 解題者指南 (Solver Guide)

## 為什麼要來這裡解題？

你的 AI 作品集沒人看，因為僱主分不出哪些是你做的、哪些是 AI 做的。

在 SolveTW：
- **解的是真實企業的真實問題** — 不是課堂作業
- **有業界主管認真看你的作品並給評語** — 不是只有老師打分
- **你的 Decision Log 紀錄判斷力** — AI 無法取代的部分
- **聲望累積到一定程度，企業可以主動聯繫你**

最差的情況：你練習了真實世界的問題解決能力。
最好的情況：你畢業前拿到業界推薦。

---

## 完整流程

```
1. 瀏覽題目 → 2. 報名 → 3. 解題 → 4. 撰寫 Decision Log → 5. 提交 PR → 6. 收到 Review → 7. 改進（選填）
```

### Step 1: 瀏覽題目

去 [problems/](../problems/) 瀏覽所有題目，或用 Issue 標籤篩選：

- **難度**：`difficulty: beginner` / `intermediate` / `advanced` / `expert`
- **產業**：`industry: 製造業` / `零售服務` / `教育` / ...
- **類型**：`type: quick-fix` / `sprint` / `project` / `capstone`
- **狀態**：`status: open`（接受報名）

**選題建議**：
- 第一次？選 ⭐ Beginner 的 Quick Fix，先跑完一次完整流程
- 選你有興趣的產業，不要只看難度
- 看驗收標準 — 越具體越好做

### Step 2: 報名

在題目 Issue 下留言報名，或用 [報名表單](../../issues/new?template=apply-to-solve.yml)。

報名時需要寫：
- 個人或團隊
- 初步想法（幾句話就好）
- 你的相關技能
- 預計完成時間

### Step 3: 解題

#### 工作目錄

Fork 這個 repo，在以下路徑建立你的解法目錄：

```
problems/[題目編號]-[題目名稱]/solutions/[你的名字或團隊名]/
├── README.md           ← 解法說明（怎麼用、怎麼跑）
├── DECISION-LOG.md     ← Decision Log（必填！）
├── src/                ← 程式碼
├── demo/               ← Demo 截圖或連結
└── tests/              ← 測試（如果有的話）
```

#### 解題心法

1. **先理解問題，再動手** — 不要一拿到題目就開始寫 code
2. **用 AI，但記錄你的判斷** — AI 建議了什麼、你採納或拒絕了什麼
3. **隨手記 notes** — 解題過程中開一個 `notes.md`，最後整理成 Decision Log
4. **測試你的解法** — 跑一下看看結果符不符合驗收標準
5. **寫清楚怎麼使用** — 想像一個陌生人要跑你的程式

#### 可以用 AI 嗎？

**完全可以。** SolveTW 鼓勵使用 AI。

但你必須在 Decision Log 中記錄：
- 用了哪些 AI 工具
- AI 做了什麼、你做了什麼
- 你在哪些地方做出了跟 AI 不同的判斷

**這不是在抓作弊。** 這是在記錄你的判斷力。
用 AI 用得好，本身就是一種能力。但你得讓 Reviewer 看到你在哪裡思考。

### Step 4: 撰寫 Decision Log

**這是最重要的步驟。** Decision Log 比程式碼更能決定你的聲望。

複製模板：[Decision Log Template](decision-log-template.md)
撰寫指南：[Decision Log 撰寫指南](decision-log-guide.md)

六個必填欄位：
1. **問題理解** — 用自己的話重述問題（至少 100 字）
2. **方案探索** — 考慮過的方案（至少 2 個）
3. **AI 使用紀錄** — 工具 + AI/人類分工 + 關鍵決策點
4. **取捨紀錄** — 犧牲了什麼
5. **失敗紀錄** — 失敗的嘗試（至少 1 筆）
6. **複用指南** — 後人怎麼用

> 沒有 Decision Log 或品質太差，聲望積分直接砍半。

### Step 5: 提交 PR

1. 確認你的目錄結構正確
2. 確認 Decision Log 六個欄位都有填
3. 建立 Pull Request，使用 [Solution Submission 模板](../.github/PULL_REQUEST_TEMPLATE/solution-submission.md)
4. 在 PR 描述中對照驗收標準自評

### Step 6: 收到 Review

Reviewer 會在你的 PR 上留下：
- 各維度評分（1-5 分）
- 具體回饋（做得好的 + 可以改進的）

**收到 Review 後**：
- 認真讀回饋，不要只看分數
- 如果 Reviewer 給了改進建議，考慮修改後重新提交
- 如果被 Reject，不要灰心 — 根據建議改進，重新提交

### Step 7: 改進（選填）

根據 Review 回饋修改後，push 新的 commit 到同一個 PR。
Reviewer 會重新看。改進後被 Approve 的解法，聲望不會比一次通過的少。

## 聲望怎麼算

```
總聲望 = 解題分(35%) + Review權重分(30%) + 複用分(20%) + 貢獻分(15%)
```

**關鍵**：不是做得多就聲望高，而是被「對的人」認可。
業界主管的一個 Approve，抵得上十個同學互評。

完整規則：[聲望計算規則](reputation-rules.md)

### 聲望等級

| 等級 | 聲望 | 解鎖 |
|------|------|------|
| 🌱 Newcomer | 0-49 | 報名解題、提交解法 |
| 🔧 Contributor | 50-199 | + Discussion、Fork 解法 |
| 🏗️ Builder | 200-499 | + 受邀 Review、建議出題 |
| ⚡ Expert | 500-999 | + 申請認證評審、優先報名 |
| 🏆 Master | 1000+ | + 被推薦給企業、開放聯繫 |

## 隱私保護

- **你預設是匿名的** — 用暱稱參與，不強制揭露學校
- **你控制揭露程度** — 什麼時候、對誰揭露真實身份，由你決定
- **聲望先行** — 僱主先看到你的能力數據，不是學校名稱

這個設計是刻意的：**先讓能力說話，再揭露身份。**

## 常見問題

### 「我程度很差，會不會丟臉？」

不會。SolveTW 不是比賽，沒有排名壓力。
而且你是匿名的，沒人知道你是誰。

選 ⭐ Beginner 的題目開始，跑完一次完整流程，你就有信心了。

### 「AI 做了大部分，我的貢獻在哪？」

你的貢獻在 Decision Log 裡：
- 你**選擇**了正確的問題切入角度
- 你**判斷**了 AI 的建議哪些該採納
- 你**發現**了 AI 的錯誤並修正
- 你**決定**了取捨

這些就是你的價值。AI 是工具，你是駕駛。

### 「我可以 Fork 別人的解法來改嗎？」

可以！但必須：
1. 在 Decision Log 中引用來源
2. 說明你做了什麼修改、為什麼
3. 原作者會獲得複用分

站在前人肩膀上，是 SolveTW 鼓勵的。

### 「團隊解題怎麼分工？」

在 Decision Log 中說明分工。每個人負責的部分都要有自己的 AI 使用紀錄。
團隊成員共享聲望（均分）。

### 「解法被 Reject 了怎麼辦？」

正常。Reject 附帶 Review 建議，根據建議改進後重新提交。
被 Reject 再通過的解法，聲望不會比一次通過的少。
重點是你展現了「接受回饋、持續改進」的能力。
