# 出題者指南 (Problem Owner Guide)

## 你需要做什麼

| 步驟 | 花費時間 | 做什麼 |
|------|---------|--------|
| 1. 出題 | 20-30 分鐘 | 填寫出題表單 |
| 2. 等待 | 0 分鐘 | 學生報名、解題（你不用管） |
| 3. Review | 30-120 分鐘 | 看解法、給評語 |

**總共花費：不到 2 小時。**

---

## Step 1: 出題

### 點這裡出題

→ [新增題目](../../issues/new?template=new-problem.yml)

### 什麼是好題目？

**好題目的特徵**：
- ✅ 來自真實的業務痛點（不是虛構的）
- ✅ 有明確的目標和驗收標準
- ✅ 抽象化到可以公開的程度（不含商業機密）
- ✅ 學生用免費工具就能做
- ✅ 有教育價值 — 解題者能學到東西

**不好的題目**：
- ❌ 「幫我做一個完整的 ERP 系統」— 太大、像免費外包
- ❌ 「做得好就通過」— 沒有可量化的驗收標準
- ❌ 「需要用到我們公司的內部資料庫」— 無法脫敏
- ❌ 「幫我寫一篇報告」— 沒有技術挑戰

### 題目範例

#### 好題目範例：⭐⭐ Intermediate

**標題**：設計 AI 客訴自動分類系統

**背景**：我們是中型製造廠，每月約 500 張客訴。目前人工分類每張 10 分鐘，錯誤率 15%。

**目標**：
1. 自動將客訴文字分為 5 個類別
2. 準確率 > 85%
3. 單次處理 < 30 秒

**驗收標準**：
1. [ ] 接收文字輸入，輸出分類結果
2. [ ] 在 50 筆測試資料上準確率 ≥ 85%
3. [ ] 有使用說明文件
4. [ ] 提供可運行的 Demo

**可提供資料**：100 筆脫敏客訴範例（CSV）

**限制**：不能用付費 API

### 如何抽象化問題

你的真實問題可能包含商業機密。做法是「抽象化」：

| 真實問題 | 抽象化後 |
|---------|---------|
| 「我們的 A 客戶流失率很高」 | 「設計客戶流失預警模型」 |
| 「倉庫 B 區的庫存經常算錯」 | 「設計 AI 輔助庫存盤點流程」 |
| 「業務部每天花 2 小時整理 SAP 報表」 | 「自動化 ERP 報表彙整」 |

保留問題的本質，去掉具體的客戶名、金額、內部流程細節。

---

## Step 2: 等待

學生報名後會在你的 Issue 下留言。你可以：
- 回答學生的問題
- 如果題目需要脫敏資料，核准後再提供

不需要頻繁關注，設定 GitHub 通知即可。

---

## Step 3: Review

### 你會收到什麼

學生提交 PR（Pull Request）時，你會收到通知。PR 中包含：
- 程式碼 / 系統原型
- **Decision Log** — 解題者的完整思考過程
- Demo 或截圖
- 驗收標準自評

### 怎麼 Review

**請重點看 Decision Log，而不只是程式碼。**

問自己：
1. 他真的理解問題嗎？還是只看到表面？
2. 他考慮過替代方案嗎？選擇的理由合理嗎？
3. AI 做了多少？他在哪些地方展現了自己的判斷？
4. 他知道自己的方案犧牲了什麼嗎？
5. 他對失敗誠實嗎？

### Review 評分表

在 PR 上留言，格式如下：

```markdown
## Review

**整體評分**: ⭐⭐⭐⭐ (4/5)

### 評分明細

| 維度 | 分數 (1-5) | 評語 |
|------|-----------|------|
| 問題理解 | 4 | 有抓到核心問題，假設合理 |
| 方案選擇 | 5 | 比較了 3 個方案，分析很到位 |
| AI 使用透明度 | 4 | 有記錄關鍵決策點 |
| 技術實作 | 3 | 可以動，但錯誤處理需加強 |
| Decision Log 品質 | 4 | 失敗紀錄很誠實 |
| 實用性 | 3 | 概念可行，但離實際部署還有距離 |

### 具體回饋

**做得好的地方**:
- ...

**可以改進的地方**:
- ...

### 這個人適合我們公司嗎？（選填，僅供內部參考）

[ ] 想進一步認識
[ ] 有潛力但還需要磨練
[ ] 能力還不足
```

### Review 的價值

你的 Review 對學生來說價值巨大：
- 一般大學生可能從來沒有被業界主管認真看過作品
- 你的 30 分鐘可能改變一個人的職涯方向
- 如果你在 Review 中留下聯繫意願，平台會通知學生

## 常見問題

### 「解法品質很差怎麼辦？」

正常。這是學習過程。給建設性的回饋比打低分更有價值。
你可以在 Review 中指出具體的改進方向，學生可以修改後重新提交。

### 「解法做得很好，我想採用怎麼辦？」

太棒了！
1. 在 PR 上標記為 `adopted`
2. 解題者會獲得大幅聲望加成
3. 如果你願意，可以給解題者一封推薦信

### 「我不想公開公司名稱」

完全沒問題。出題表單中的公司名稱是選填的，你可以用「某製造業公司」代替。

### 「我擔心學生拿我的問題描述去當作品集」

題目本身是 CC BY-NC 授權，學生不能拿去商用。
而且題目是抽象化的版本，不包含你的商業機密。

### 「我沒有 GitHub 帳號」

Phase 1 需要 GitHub 帳號（免費註冊，5 分鐘）。
Phase 2 之後會有獨立的 Web 介面，不需要 GitHub。
