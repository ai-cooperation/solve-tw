# Decision Log 撰寫指南

## 為什麼 Decision Log 是必填的？

在 vibe coding 時代，程式碼本身不再稀缺 — AI 都能寫。真正稀缺的是**判斷力**：

- 為什麼選這個方案而不是那個？
- AI 給了你什麼建議，你採納了哪些、拒絕了哪些？
- 你犧牲了什麼來換取什麼？
- 你從失敗中學到了什麼？

**Decision Log 就是你判斷力的證據。** 它比程式碼更能證明你的能力。

> 沒有 Decision Log 的解法，聲望積分直接砍半。

---

## 六個必填欄位怎麼寫

### 1. 問題理解 — 「你真的懂問題嗎？」

**好的寫法**：用自己的話重述，加上你的洞察
```
這個問題表面上是要做客訴分類，但我認為核心挑戰其實是——
客訴的文字表達很不統一，同一個問題可能有 20 種描述方式。
我的假設是：如果能先做「語義標準化」，分類準確率就能大幅提升。
```

**不好的寫法**：複製貼上題目描述
```
題目要求我們做一個客訴分類系統，準確率 > 85%。
```

### 2. 方案探索 — 「你不只會一招」

至少列出 2 個替代方案，就算你很確定要用方案 A，也要認真思考方案 B 為什麼不選。

**好的比較**：
| 方案 | 優點 | 缺點 | 選/不選 |
|------|------|------|---------|
| Few-shot Prompting | 簡單快速、不需訓練資料 | 分類數多時 prompt 太長 | **選** — 題目只有 5 個分類 |
| Fine-tuning | 準確率可能更高 | 需要大量標註資料、成本高 | 不選 — 沒有足夠訓練資料 |
| RAG + 分類 | 可以處理新類別 | 架構複雜、over-engineering | 不選 — 類別固定，不需要 |

**不好的比較**：
| 方案 | 優點 | 缺點 |
|------|------|------|
| 我的方案 | 好 | 無 |
| 其他方案 | 不好 | 很多 |

### 3. AI 使用紀錄 — 「透明是加分，不是扣分」

SolveTW 鼓勵使用 AI。但我們要求你**誠實記錄**：

**最有價值的內容**是「關鍵的人類決策點」：

```markdown
### 關鍵決策點

1. Claude 建議用 LangChain 框架，但我選擇直接呼叫 API。
   原因：這個任務很簡單，LangChain 引入不必要的複雜度。

2. Cursor 自動生成了一段錯誤處理邏輯，把所有 API 錯誤都 retry。
   我修改為：只 retry 429/503，其他錯誤直接報錯。
   原因：retry 500 可能導致重複操作。

3. AI 生成的 prompt 用英文，我改成中文。
   原因：客訴原文是中文，用中文 prompt 分類效果更好（實測準確率 +12%）。
```

這些決策點就是你區別於「只會按 Enter」的證據。

### 4. 取捨紀錄 — 「成熟的工程師知道自己犧牲了什麼」

每個方案都有代價。不存在「沒有缺點」的解法。

```markdown
- 為了快速實作（3 天完成），犧牲了可擴展性（新增分類需要改 prompt）
- 為了零成本（用免費 API），犧牲了回應速度（每次 3-5 秒）
- 已知限制：只測試了繁體中文，簡體中文和英文未測試
```

### 5. 失敗紀錄 — 「誠實 > 完美」

如果你寫「一切順利」，Reviewer 不會相信。每個專案都有意外。

```markdown
| # | 嘗試 | 預期 | 實際 | 學到什麼 |
|---|------|------|------|---------|
| 1 | 用 zero-shot prompting | 準確率 > 80% | 只有 62% | 不給範例，AI 不知道分類標準 |
| 2 | 一次送整段客訴文字 | 正常分類 | 常常分到「其他」 | 文字太長，關鍵資訊被稀釋 |
| 3 | 先摘要再分類（兩步驟） | 更準確 | 準確率到 89% | 拆步驟效果好但 latency 翻倍 |
```

### 6. 複用指南 — 「讓你的 Token 不白花」

想像一個陌生人看到你的解法，想拿去用在自己的場景。他需要知道什麼？

```markdown
### 適用場景
- ✅ 文字分類任務，類別 ≤ 10 個
- ✅ 有明確的分類定義和範例

### 不適用場景
- ❌ 類別 > 20 個（prompt 太長）
- ❌ 需要即時回應 < 1 秒（兩步驟有 latency）

### 要改什麼
| 地方 | 目前設定 | 改成什麼 |
|------|---------|---------|
| `config.py` 的 CATEGORIES | 5 個客訴類別 | 換成你的分類 |
| `prompts/classify.txt` 的範例 | 客訴範例 | 換成你的領域範例 |
| API key | Claude | 也可換 GPT-4o |
```

## 常見問題

### 「AI 做了 90%，我的 Decision Log 會不會很空？」

不會。即使 AI 做了 90% 的程式碼，你的判斷力仍然體現在：
- **選擇用哪個 AI 工具**（為什麼選 Claude 不選 GPT？）
- **給 AI 什麼指令**（Prompt 設計本身就是判斷）
- **決定要不要採納 AI 的建議**（你拒絕了什麼？）
- **測試和修正**（AI 生成的東西你怎麼驗證？）

### 「寫 Decision Log 要花多久？」

通常 30-60 分鐘。如果你在解題過程中就隨手記錄（強烈建議），最後整理只需要 20 分鐘。

**建議**：解題時開一個 `notes.md`，隨手記下每個重要決策。最後再整理成正式的 Decision Log。

### 「失敗紀錄真的不會扣分嗎？」

不會。誠實的失敗紀錄反而**加分**。Reviewer 的 Rubric 明確規定：

- 🥇 Exemplary：「誠實、具體、有學習反思」
- ❌ 扣分：「空白或『一切順利』」

### 「可以看別人的 Decision Log 當參考嗎？」

可以！這正是 SolveTW 的精神 — 站在前人肩膀上。
但你的 Decision Log 必須是你自己的思考過程，不能複製別人的。
